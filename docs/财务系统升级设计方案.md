## 1. 背景与目标

当前系统的财务模块主要围绕「收入 / 支出账单」展开，适合记录个人和家庭的日常收支。

接下来希望增加以下能力：

1. 支持多币种。
2. 除了收入、支出外，增加换汇操作。
3. 增加“桶”余额，例如微信中有多少钱。
4. 增加收入、支出曲线图。
5. 增加投资收益和亏损曲线图，并重新思考投资数据是否应该独立建模。
6. 解决家庭多用户下账户、分类、桶 ID 不一致时的聚合问题。
7. 确认收入 / 支出不再单独拆分具体操作类型，统一由分类承担统计维度。

这些需求本质上不是简单给账单表增加几个字段，而是把系统从：

```text
单币种收入 / 支出流水
```

升级为：

```text
多币种资产流水 + 账户余额 + 投资资产表现
```

因此设计上需要从「账单记录」升级到「资金流 + 资产快照」。

```text
账单记录：记录我花了什么、赚了什么
资金流：记录钱从哪里来、到哪里去
资产快照：记录我现在拥有什么、价值多少
```

## 3. 需求一：增加多币种

### 3.1 核心问题

多币种后，一条交易至少会涉及两个金额概念：

```text
原始金额：这笔交易实际发生的币种金额
折算金额：为了统计统一换算成基准币种后的金额
```

例如：

```text
收入：100 USD
当日汇率：1 USD = 7.20 CNY
折算金额：720 CNY
```

### 3.2 建议字段

交易表建议增加：

```text
amount          原始金额
currency        原始币种，例如 CNY / USD / HKD
base_amount     折算到基准币种后的金额
base_currency   用户基准币种，默认 CNY
exchange_rate   交易时使用的汇率
```

### 3.3 基准币种

决策：

```text
currency 是全平台公共字典，所有用户共用。
base_currency 是统计基准币种，用户和家庭各自独立设置。
个人和家庭都支持随时更改 base_currency。
报表等视图根据当前用户的 base_currency 或当前家庭的 base_currency 动态展示。
```

用户基准币种：

```text
user.base_currency
```

家庭基准币种：

```text
family.base_currency
```

统计逻辑：

```text
个人账单：折算为用户 user.base_currency 展示
家庭账单：折算为 family.base_currency 展示
切换基准币种时，历史交易根据保存的汇率快照重新折算
```

影响范围：

```text
首页总资产
收入支出统计
家庭成员账单统计
收支趋势图
分类占比
年度 / 月度分析
```

当前项目默认值：

```text
user.base_currency 默认 CNY
family.base_currency 默认 CNY
```

### 3.4 币种配置

如果第一阶段只支持常见币种，可以先用配置或枚举：

```text
CNY 人民币
USD 美元
HKD 港币
EUR 欧元
JPY 日元
GBP 英镑
SGD 新币
```

如果希望更正式，可以设计 `currency` 表：

```text
currency
  id
  code        CNY / USD / HKD
  name        人民币 / 美元
  symbol      ¥ / $ / HK$
  precision   2
  enabled
```

### 3.5 汇率表与汇率获取策略

决策：

```text
使用免费每日汇率 API 自动获取
优先使用本地 exchange_rate 缓存
允许用户手动修改汇率
交易表保存最终使用的汇率快照
```

建议单独保存汇率：

```text
exchange_rate
  id
  from_currency
  to_currency
  rate
  rate_date
  source
  created_at
```

汇率获取流程：

```text
1. 用户新增多币种交易。
2. 系统根据交易日期、原币种、目标基准币种自动查汇率。
3. 如果本地 exchange_rate 表已有当天汇率，直接使用。
4. 如果没有，则调用免费汇率 API 获取。
5. 获取成功后写入 exchange_rate 表。
6. 获取失败时，让用户手动输入汇率。
7. 用户可以手动修正系统获取的汇率。
8. financial_event 保存最终使用的 exchange_rate 快照。
```

原因：

```text
历史交易的折算金额不能因为今天汇率变了而变化。
不同来源的汇率口径不同，用户应能按真实成交汇率修正。
```

第一阶段可选免费 API：

```text
Frankfurter：免费，无需 API Key，适合常见法币和每日汇率。
```

后续如果需要实时汇率、高频调用、商业级稳定性或更完整历史数据，再考虑付费汇率源。

### 3.6 多币种统计原则

统计可以分为两种方式。

#### 方式 A：按原币种展示

```text
CNY 收入：10,000
USD 收入：500
HKD 收入：1,000
```

优点：真实。  
缺点：不好合计。

#### 方式 B：全部折算成基准币种

```text
总收入：17,200 CNY
```

优点：适合总览、趋势图、家庭统计。  
缺点：依赖汇率。

### 3.7 推荐方案

默认使用：

```text
折算成基准币种后的金额
```

同时提供切换能力：

```text
按原币种查看
```

## 4. 需求二：增加换汇操作

### 4.1 为什么换汇不能算收入或支出

例如人民币换美元：

```text
支出 720 CNY
收入 100 USD
汇率 7.2
手续费 10 CNY
```

如果把它记成普通收入和支出，统计图会错误地认为这个月收入和支出都增加了。

但实际上，换汇只是资产形态变化：

```text
人民币资产减少
美元资产增加
```

所以换汇应该是独立交易类型，不能进入普通收入支出统计。

### 4.2 交易类型扩展

当前系统：

```text
1 = 收入
2 = 支出
```

决策：第一版交易类型确认如下：

```text
income                收入
expense               支出
refund                退款
transfer              转账
exchange              换汇
receivable_create     形成应收
receivable_collect    收回应收
deposit_create        支付押金
deposit_refund        退回押金
loan_out              借出
loan_collect          收回借出款
investment_buy        投资买入
investment_sell       投资卖出
investment_income     投资收入
balance_adjustment    余额调整
```

统计规则：

```text
income / expense 默认进入普通收入支出统计
refund 用于抵扣支出
transfer / exchange / receivable / deposit / loan / investment_buy / investment_sell / balance_adjustment 默认不进入普通收入支出统计
investment_income 只进入投资统计，不进入普通收入统计
```

退款规则已确认：退款单独使用 `refund` 类型，不记成普通收入。

示例：

```text
原支出：
  event_type = expense
  amount = 100
  category = 购物
  ledger_entry = 工商银行-CNY -100

退款：
  event_type = refund
  amount = 30
  category = 购物
  ledger_entry = 工商银行-CNY +30
```

报表统计：

```text
购物支出净额 = 100 - 30 = 70
收入不增加
```

如果退款能关联原始支出，建议使用 `event_group_id` 或 `related_financial_event_id` 建立关联；如果无法关联，也可以作为独立 refund 记录处理。

余额调整规则已确认：允许用户手动创建 `balance_adjustment`，但必须填写备注，默认不进入普通收入 / 支出统计。

适用场景：

```text
历史数据缺失
导入数据不完整
实际余额和系统余额不一致
初始化 Bucket 余额后需要校正
```

示例：

```text
系统微信零钱余额：1000
实际微信零钱余额：980

event_type = balance_adjustment
remark = 修正微信零钱实际余额
ledger_entry = 微信零钱-CNY -20
include_in_statistics = false
```

余额调整只改变 Bucket 余额，不影响普通收入 / 支出报表。

### 4.3 换汇的数据结构

换汇是一种双边交易：

```text
从一个桶扣除一种币种
向另一个桶增加另一种币种
```

例如：

```text
从 招行-CNY 扣除 7,200 CNY
向 招行-USD 增加 1,000 USD
汇率：7.2
手续费：10 CNY
```

建议换汇详情字段：

```text
event_type = exchange

from_bucket_id
from_currency = CNY
from_amount = 7200

to_bucket_id
to_currency = USD
to_amount = 1000

exchange_rate = 7.2
fee_amount = 10
fee_currency = CNY
fee_bucket_id = 招行-CNY
```

### 4.4 换汇对统计的影响

决策规则：

```text
换汇本金不影响收入 / 支出统计
手续费是可选的，不是所有换汇都有手续费
无手续费：只生成 exchange financial_event
有手续费：生成 exchange financial_event + expense financial_event
手续费支出计入普通支出统计
汇兑损益可以单独统计，但第一阶段可以不做
```

有手续费时，推荐把手续费单独生成一笔支出交易，并用 `event_group_id` 和换汇交易关联。

示例：

```text
event_group_id = EXCHANGE_20260610_001

FinancialEvent A:
  event_type = exchange
  include_in_statistics = false

FinancialEvent B:
  event_type = expense
  category = 金融手续费
  amount = 10 CNY
  include_in_statistics = true
```

这样普通支出统计可以看到金融手续费，同时换汇详情页也能通过 `event_group_id` 展示本次换汇关联的手续费。

### 4.5 转账手续费规则

决策：转账手续费和换汇手续费使用同一规则。

```text
转账本金不影响收入 / 支出统计
手续费是可选的，不是所有转账都有手续费
无手续费：只生成 transfer financial_event
有手续费：生成 transfer financial_event + expense financial_event
手续费支出计入普通支出统计
```

有手续费时，手续费单独记录成一笔支出交易，并用 `event_group_id` 和转账交易关联。

示例：

```text
event_group_id = TRANSFER_20260610_001

FinancialEvent A:
  event_type = transfer
  include_in_statistics = false

LedgerEntry A1:
  bucket = 微信零钱-CNY
  direction = out
  amount = 1000

LedgerEntry A2:
  bucket = 招商银行-CNY
  direction = in
  amount = 1000

FinancialEvent B:
  event_type = expense
  category = 金融手续费
  amount = 2 CNY
  include_in_statistics = true

LedgerEntry B1:
  bucket = 微信零钱-CNY
  direction = out
  amount = 2
```

说明：

```text
转账本金只是资产位置变化，不算支出。
手续费是真实成本，算支出。
转账和手续费通过 event_group_id 关联，方便详情页合并展示。
```

## 5. 需求三：增加“桶”余额

### 5.1 问题：桶要不要和账户关联？

决策：已确认。

```text
Account = 平台 / 机构 / 账户分组
Bucket = 真正的余额容器
```

账户 Account 和桶 Bucket 分开，但 Bucket 应该关联 Account。

即：

```text
Account 1 : N Bucket
```

### 5.2 为什么账户和桶不能完全等同

因为一个账户可能有多个币种余额。

例如：

```text
招商银行账户
  ├── CNY 余额
  ├── USD 余额
  └── HKD 余额
```

微信也可能有多个资金池：

```text
微信
  ├── 零钱
  ├── 零钱通
  └── 银行卡快捷支付
```

因此：

```text
Account 表示平台 / 机构
Bucket 表示具体资金池 / 余额容器
```

### 5.3 Account 示例

```text
微信
支付宝
招商银行
工商银行
现金
IBKR
老虎证券
富途
```

决策：Account 不保存余额。

```text
Account 只表示平台 / 机构 / 分组
余额全部保存在 Bucket
Account 总余额通过其下 Bucket 汇总得到
```

示例：

```text
微信 Account
  微信零钱 Bucket：100
  零钱通 Bucket：1000

微信 Account 总余额 = 1100
```

这样可以避免 `Account.balance`、`Bucket.balance`、`LedgerEntry` 三套余额来源互相不一致。

### 5.4 Bucket 示例

```text
微信-零钱-CNY
支付宝-余额-CNY
招商银行-活期-CNY
招商银行-美元现汇-USD
IBKR-现金-USD
现金钱包-CNY
AA应收款-CNY
租房押金-CNY
沪深300基金-CNY
信用卡欠款-CNY
```

### 5.5 Bucket 表建议

```text
bucket
  id
  user_id
  account_id
  name
  currency
  balance
  initial_balance
  bucket_type
  bucket_nature
  is_active
  sort
  created_at
  updated_at
```

决策：Bucket 是 Financial App 的核心模型，表示资产、负债或虚拟资金池，不只是账户余额。

Bucket 类型已确认：第一版使用下面这些类型，能够覆盖当前场景。

`bucket_type` 可选值：

```text
cash              现金
wallet            电子钱包
bank              银行账户
credit            信用卡
investment_cash   投资账户现金
investment_asset  投资资产
receivable        应收款
deposit           押金 / 保证金
loan_out          借出款
liability         负债
virtual           虚拟桶
```

`bucket_nature` 用于区分资产和负债：

```text
asset      资产
liability  负债
```

推荐映射：

| bucket_type | bucket_nature |
|---|---|
| cash | asset |
| wallet | asset |
| bank | asset |
| investment_cash | asset |
| investment_asset | asset |
| receivable | asset |
| deposit | asset |
| loan_out | asset |
| credit | liability |
| liability | liability |
| virtual | asset 或 liability，按具体用途设置 |

净资产计算：

```text
净资产 = asset 类型 Bucket 余额合计 - liability 类型 Bucket 余额合计
```

### 5.6 余额维护方案

#### 方案 A：直接存当前余额

每次交易后更新 `bucket.balance`。

优点：

```text
查询快，实现简单
```

缺点：

```text
历史修正、删除交易、导入重复时容易导致余额错乱
```

#### 方案 B：通过流水计算余额

```text
当前余额 = 初始余额 + 所有流入 - 所有流出
```

优点：

```text
准确、可追溯
```

缺点：

```text
查询复杂，性能稍高
```

#### 推荐：混合模式

```text
bucket.balance 保存当前余额快照
financial_event / ledger_entry 保存所有变动明细
```

即：

- 平时查询直接看 `bucket.balance`
- 每次交易都产生余额变动流水
- 必要时可以通过流水重算余额
- 后台可以做余额校验

### 5.7 编辑 / 删除交易时的余额处理

决策：第一阶段采用“回滚旧 LedgerEntry，再重建新 LedgerEntry”的方式。

编辑交易时：

```text
1. 开启数据库事务。
2. 查询旧 financial_event 对应的所有 ledger_entry。
3. 按旧 ledger_entry.amount 的反方向回滚 Bucket.balance。
4. 删除或软删除旧 ledger_entry。
5. 更新 financial_event。
6. 根据新的交易内容生成新的 ledger_entry。
7. 根据新的 ledger_entry.amount 更新 Bucket.balance。
8. 提交事务。
```

删除交易时：

```text
1. 开启数据库事务。
2. 查询旧 financial_event 对应的所有 ledger_entry。
3. 按旧 ledger_entry.amount 的反方向回滚 Bucket.balance。
4. 删除或软删除 ledger_entry。
5. 删除或软删除 financial_event。
6. 提交事务。
```

示例：

```text
原支出：微信零钱 -50
编辑为：微信零钱 -80

处理过程：
  先回滚旧流水：微信零钱 +50
  再生成新流水：微信零钱 -80
```

这样可以保证：

```text
FinancialEvent、LedgerEntry、Bucket.balance 始终一致
```

第一阶段不采用“反冲流水”方案，避免流水过多、实现复杂。后续如果需要更强审计能力，可以升级为反冲模式。

### 5.8 事务与并发控制

决策：所有会影响 Bucket 余额的操作，必须使用数据库事务，并锁定相关 Bucket 行。

适用操作：

```text
新增交易
编辑交易
删除交易
转账
换汇
应收形成 / 收回
押金支付 / 退回
投资买入 / 卖出
余额调整
```

事务内必须同时完成：

```text
创建 / 更新 financial_event
创建 / 删除 / 重建 ledger_entry
更新 bucket.balance
```

如果任一步失败，必须整体回滚。

更新 Bucket 前需要锁定相关 Bucket，例如 MySQL：

```sql
SELECT * FROM bucket WHERE id IN (...) FOR UPDATE;
```

余额更新建议使用增量更新：

```sql
UPDATE bucket
SET balance = balance + ?
WHERE id = ?;
```

其中 `?` 使用 `ledger_entry.amount`，可以是正数，也可以是负数。

这样可以避免并发场景下出现：

```text
交易记录已创建但余额没更新
一个 Bucket 扣款成功，另一个 Bucket 入账失败
两笔并发交易同时修改同一个 Bucket 导致余额错乱
```

### 5.9 Bucket 余额是否允许为负

决策：按 Bucket 类型决定是否允许负数，不单独增加 `allow_negative` 字段。

规则：

```text
cash / wallet / bank：不允许负数
receivable / deposit / loan_out：不允许负数
investment_cash / investment_asset：不允许负数
credit / liability：负债类 Bucket，balance 表示“欠款金额”，正常按正数保存
virtual：按具体用途判断，第一阶段尽量少用
```

示例：

```text
微信零钱 100，支出 150 -> 不允许
信用卡欠款 0，刷卡 150 -> 允许，balance 变 150
信用卡欠款 150，还款 200 -> 不允许，因为不能还超过欠款
AA应收 50，收回 100 -> 不允许，因为不能收超过应收
```

核心原则：

```text
资产类 Bucket 默认不允许扣成负数
负债类 Bucket 的 balance 表示欠款金额，不使用负数保存欠款
```

### 5.10 Bucket 初始余额

决策：Bucket 创建时设置 `initial_balance`，同时生成一条初始化 LedgerEntry。

不建议只在 `bucket.initial_balance` 中保存数字，因为这样余额来源不可追溯。初始化余额也应该走统一的 FinancialEvent + LedgerEntry 流程。

创建 Bucket 时：

```text
bucket.initial_balance = 用户填写的初始余额
bucket.balance = initial_balance
创建 event_type = balance_adjustment
创建 ledger_entry.amount = initial_balance
remark = 初始化余额
```

示例一：创建工商银行 Bucket。

```text
创建 工商银行-CNY
初始余额：10000

financial_event:
  type = balance_adjustment
  remark = 初始化余额
  include_in_statistics = false

ledger_entry:
  bucket = 工商银行-CNY
  amount = +10000
```

示例二：创建信用卡欠款 Bucket。

```text
创建 信用卡-CNY
初始欠款：2000

financial_event:
  type = balance_adjustment
  remark = 初始化信用卡欠款
  include_in_statistics = false

ledger_entry:
  bucket = 信用卡-CNY
  amount = +2000
```

说明：

```text
信用卡 Bucket 的 balance 表示欠款金额。
+2000 表示信用卡欠款增加到 2000。
```

这样后续可以通过 LedgerEntry 重算 Bucket 余额：

```text
理论余额 = 所有 ledger_entry.amount 合计
```

### 5.11 是否需要 Ledger 表

如果系统要做余额，建议增加 `ledger_entry` 表。

原因：收入、支出、换汇、转账、投资买卖，本质上都会改变某个桶的余额。

可以设计成：

```text
financial_event 主表：记录业务事件
ledger_entry 明细表：记录每个桶的余额变化
```

决策：LedgerEntry 使用带正负号的金额表示 Bucket 余额变化，不再使用 `direction = in/out`。

```text
amount > 0：这个 Bucket 的余额增加
amount < 0：这个 Bucket 的余额减少
```

一笔支出示例：

```text
financial_event:
  type = expense
  description = 吃饭

ledger_entry:
  bucket = 微信-CNY
  amount = -50
  currency = CNY
```

一笔收入示例：

```text
financial_event:
  type = income
  description = 工资

ledger_entry:
  bucket = 招行-CNY
  amount = +20000
```

一笔换汇示例：

```text
financial_event:
  type = exchange

ledger_entry 1:
  bucket = 招行-CNY
  amount = -7200
  currency = CNY

ledger_entry 2:
  bucket = 招行-USD
  amount = +1000
  currency = USD
```

AA 应收示例：

```text
形成应收：
  工商银行-CNY amount = -150
  AA应收款-CNY amount = +150

收回应收：
  AA应收款-CNY amount = -50
  微信零钱-CNY amount = +50
```

押金示例：

```text
支付押金：
  工商银行-CNY amount = -3000
  押金-CNY amount = +3000

退回押金：
  押金-CNY amount = -3000
  工商银行-CNY amount = +3000
```

基金定投示例：

```text
工商银行-CNY amount = -100
基金持仓-CNY amount = +100
```

信用卡示例：

```text
信用卡消费：
  信用卡欠款-CNY amount = +100

还信用卡：
  工商银行-CNY amount = -100
  信用卡欠款-CNY amount = -100
```

说明：

```text
正负数只表示 Bucket 余额如何变化，不表示收入 / 支出语义。
收入 / 支出语义由 event_type 决定。
```

## 7. 需求五：投资收益和亏损曲线图

### 7.1 现状问题

当前投资收益和亏损是按月记录到普通收入 / 支出账单表里的。

这短期可以用，但长期不推荐。

### 7.2 为什么投资不适合继续放普通账单表

普通收入 / 支出是现金流。

例如：

```text
工资收入 +20000
吃饭支出 -50
房租支出 -5000
```

投资收益 / 亏损可能不是现金流。

例如：

```text
买入成本：10000
当前市值：12000
浮盈：2000
```

这 2000 元没有真正到账，属于：

```text
未实现收益
```

如果把它记成收入，会导致收入统计虚高。

同理，投资浮亏也不是生活支出。如果记成支出，会污染消费曲线。

### 7.3 推荐结论

建议：

```text
投资应该单独建模，不要长期混在普通收入 / 支出账单里。
```

但可以分阶段迁移，不需要一次性推翻现有数据。

### 7.4 投资模块推荐结构

建议至少分成三层：

```text
投资账户
投资资产
投资估值 / 收益快照
```

#### 7.4.1 投资账户

例如：

```text
富途
老虎
IBKR
支付宝基金
天天基金
招商证券
```

建议投资账户仍然复用 Account：

```text
account.type = investment
```

不必一开始单独再建一套账户体系。

#### 7.4.2 投资资产

表示具体投资标的。

```text
investment_asset
  id
  user_id
  account_id
  name
  code
  asset_type
  currency
  market
  created_at
  updated_at
```

示例：

```text

苹果股票
贵州茅台
比特币
沪深300基金
债券基金
```

`asset_type` 可选：

```text

etf
bond
crypto
cash
other
```

#### 7.4.3 投资快照

用于记录某一天或某个月的投资表现。

```text
investment_snapshot
  id
  user_id
  asset_id 可选
  account_id 可选
  snapshot_date
  currency
  principal_amount
  market_value
  realized_profit
```

字段含义：

```text

```

如果第一阶段不想细到每个资产，可以先做到：

```text
按投资账户每月记录总收益
```

### 7.5 投资曲线图建议

建议支持：

```text

```

示例：

```text

```

### 7.6 投资是否进入收入支出统计

决策：投资相关交易不进入普通收入 / 支出统计，只进入投资统计。

普通账单统计只看：

```text
工资、生活消费、房租、餐饮、交通等现金流
```

投资图表单独统计：

```text
投资收益、亏损、收益率、资产市值
```

投资买入 / 投资卖出的处理规则：

```text
investment_buy / investment_sell 只记录资金和投资资产 Bucket 的变化
投资收益、亏损、收益率通过 investment_snapshot 统计
investment_income 只进入投资统计，不进入普通收入统计
```

示例：

```text
买入基金 100：
  银行 Bucket -100
  基金 Bucket +100
  不计入普通支出

卖出基金：成本 100，到账 110：
  基金 Bucket -100
  银行 Bucket +110
  收益 10 进入投资统计，不进入普通收入
```

因此，普通收入支出统计默认排除所有投资类型：

```text
investment_buy
investment_sell
investment_income
投资浮盈 / 浮亏
```

### 7.7 历史投资数据迁移

当前已有投资收益 / 亏损记录在普通账单表中，不建议直接删除。

建议分三步迁移。

#### 第一步：给现有投资类账单打标签

识别：

```text
分类 = 投资收益
分类 = 投资亏损
```

或者增加：

```text
category_type = investment
```

先不改历史数据，只是在统计时能区分。

#### 第二步：新建投资快照表

后续新的投资收益 / 亏损进入：

```text
investment_snapshot
```

不再作为普通收入支出记录。

#### 第三步：迁移历史投资数据

例如原来：

```text
2026-01 投资收益 +3000
2026-02 投资亏损 -2000
```

可以迁移为：

```text
investment_snapshot
  snapshot_month = 2026-01
  total_profit = 3000

investment_snapshot
  snapshot_month = 2026-02
  total_profit = -2000
```

如果缺少本金、市值，先允许为空。

## 8. 家庭多用户聚合设计

### 8.1 核心问题

家庭记账是多用户场景。不同用户各自维护账户、分类和桶，因此它们的 ID 不一样。

例如：

```text
用户A:
  account_id = 1, 微信
  account_id = 2, 支付宝
  account_id = 3, 招商银行

用户B:
  account_id = 4, 微信
  account_id = 5, 支付宝
  account_id = 6, 工商银行
```

如果家庭统计直接按 `account_id` 聚合，会得到错误结果：

```text
account_id=1 微信
account_id=4 微信
```

系统会认为它们是两个完全不同的聚合项。

但家庭统计真正想看的通常是：

```text
家庭微信支出合计
家庭支付宝支出合计
家庭银行卡支出合计
```

所以家庭聚合不能直接按用户自己的 `account_id`、`category_id`、`bucket_id` 聚合。

核心原则：

```text
个人明细使用本地 ID。
家庭汇总使用标准化聚合维度。
```

也就是：

```text
个人账本：account_id / category_id / bucket_id
家庭账本：account_group_key / category_group_key / provider_code / account_type / bucket_type / currency
```

### 8.2 家庭资产报表不做账户共享和强制聚合

决策：不需要家庭共享账户。

```text
Account 永远属于个人用户
不支持 family account
不需要 scope = personal / family
不需要 family_id 挂在 account 上
```

家庭现有资产报表只需要把家庭成员的个人 Account / Bucket 一起展示出来。

例如：

```text
家庭资产
  张三
    微信
      微信零钱-CNY：1000
    工商银行
      工商银行-CNY：5000

  李四
    微信
      微信零钱-CNY：800
    支付宝
      支付宝余额-CNY：2000
```

也就是说，家庭资产视图的核心是：

```text
按家庭成员展示各自的 Account / Bucket
```

而不是：

```text
把所有人的微信强行聚合成一个“家庭微信”
```

家庭资产报表可以计算总资产、总负债、净资产，但明细展示仍然保留成员维度：

```text
家庭总资产 = 所有家庭成员 asset Bucket 折算后合计
家庭总负债 = 所有家庭成员 liability Bucket 折算后合计
家庭净资产 = 家庭总资产 - 家庭总负债
```

决策：家庭账单统计和家庭资产报表分开。

```text
账单：看我干了啥
资产：看我有啥
```

家庭账单统计关注一段时间内发生的事情：

```text
收入
支出
退款
分类统计
趋势图
家庭成员消费对比
钱花在哪些分类
```

决策：家庭账单统计不按 Account / Bucket 聚合。

原因：

```text
家庭账单是看干了啥
Account / Bucket 是看钱在哪里
```

家庭账单统计保留维度：

```text
按家庭成员
按收入 / 支出
按 CategoryGroup
按趋势
按时间范围
```

不做这些维度：

```text
按 Account
按 Bucket
按支付渠道
```

Account / Bucket 相关信息放在资产报表或交易明细中。

个人账单统计同理，也不按 Account / Bucket 聚合。

个人账单统计保留维度：

```text
按收入 / 支出
按 CategoryGroup
按趋势
按时间范围
```

个人账单统计不做这些维度：

```text
按 Account
按 Bucket
按支付渠道
```

Account / Bucket 相关信息同样放在资产视图或交易明细中。

核心问题：

```text
这段时间干了啥？
```

家庭资产报表关注某个时间点当前拥有或欠下的东西：

```text
现金
银行卡
微信 / 支付宝
应收款
押金
投资资产
信用卡负债
贷款
净资产
```

核心问题：

```text
现在有啥？
```

两者数据来源不同：

```text
账单统计：看 event_type + category + time range
资产报表：看 Bucket.balance + bucket_nature + currency
```

例如羽毛球 AA：

```text
真实支出 50
形成应收 150
收回应收 150
```

账单统计看到：

```text
运动支出 50
```

资产报表看到：

```text
工商银行少了 200
微信零钱多了 150
AA应收款最终为 0
净资产少了 50
```

### 8.3 Account 推荐补充字段

账户表建议补充：

```text
account
  id
  user_id
  name
  type
  provider_code
  account_group_key
  standard_account_id 可选
  bank_code 可选
  display_name
  is_active
  created_at
  updated_at
```

字段含义：

| 字段 | 含义 | 示例 |
|---|---|---|
| `id` | 用户个人账户 ID | 1 |
| `user_id` | 账户属于哪个用户 | user_a |
| `name` | 用户自己看到的名称 | 微信、工资卡、招商银行 |
| `type` | 账户大类 | wallet / bank / cash / investment |
| `provider_code` | 标准化平台编码 | WECHAT / ALIPAY / CMB |
| `account_group_key` | 家庭聚合键 | WECHAT / ALIPAY / BANK_CMB |
| `standard_account_id` | 标准账户模板 ID | 1 |
| `bank_code` | 银行编码 | CMB / ICBC |
| `display_name` | 聚合展示名 | 微信 / 支付宝 / 招商银行 |
| `scope` | 账户作用域 | personal / family |
| `family_id` | 家庭账户所属家庭 | family_a |

### 8.5 自定义账户处理

用户可能会创建标准模板无法覆盖的账户，例如：

```text
老婆小金库
家用备用金
公司报销账户
朋友代付
```

这类账户可以这样处理：

```text
provider_code = CUSTOM
account_group_key = 用户自定义聚合键
```

例如：

```text
name = 老婆小金库
provider_code = CUSTOM
account_group_key = FAMILY_RESERVE
display_name = 家庭备用金
```

不建议完全依赖账户名称聚合，因为用户可能写成：

```text
微信
微信零钱
WeChat
wx
```

这些名称语义相近，但字符串不同。

### 8.6 家庭共享账户

有些账户可能不是某个人的，而是家庭共享账户。

例如：

```text
家庭共同银行卡
家庭备用金
夫妻共同账户
家庭现金盒
```

可以考虑账户支持两种作用域：

```text
personal 个人账户
family   家庭账户
```

规则：

```text
scope = personal 时，user_id 有值，family_id 可为空
scope = family 时，family_id 有值，user_id 可为空或表示创建者
```

示例：

```text
account:
  id = 1
  scope = personal
  user_id = 张三
  name = 张三微信

account:
  id = 2
  scope = personal
  user_id = 李四
  name = 李四微信

account:
  id = 3
  scope = family
  family_id = 家庭A
  name = 家庭共同银行卡
```

这样可以同时支持：

```text
个人账户参与家庭统计
家庭账户由家庭共同管理
```

这个能力比普通聚合复杂，可以放在后续阶段。

### 8.7 Bucket 在家庭聚合中的处理

引入 Bucket 后，家庭聚合仍然不能直接按 `bucket_id` 汇总。

例如：

```text
用户A:
  Account: 微信
    Bucket: 微信零钱-CNY

用户B:
  Account: 微信
    Bucket: 微信零钱-CNY
```

两个用户的 `bucket_id` 不同，但可以共享同一组聚合维度：

```text
account.provider_code = WECHAT
bucket.currency = CNY
bucket.bucket_type = wallet
```

推荐 Bucket 补充字段：

```text
bucket
  id
  user_id
  account_id
  name
  currency
  bucket_type
  bucket_group_key
  balance
```

家庭余额统计可以支持：

#### 按成员看余额

```text
GROUP BY user_id, bucket_id
```

示例：

```text
张三：
  微信零钱 CNY 1000
  招商银行 CNY 5000

李四：
  微信零钱 CNY 800
  支付宝 CNY 2000
```

#### 按平台聚合余额

```text
GROUP BY account.provider_code, bucket.currency
```

示例：

```text
微信 CNY 总余额：1800
支付宝 CNY 总余额：2000
招商银行 CNY 总余额：5000
```

#### 按账户类型聚合余额

```text
GROUP BY account.type, bucket.currency
```

示例：

```text
电子钱包 CNY：3800
银行卡 CNY：5000
证券账户 USD：2000
现金 CNY：500
```

### 8.8 分类也需要标准聚合键

这个问题不只发生在账户上，分类也会发生。

例如：

```text
用户A:
  category_id = 1, 餐饮

用户B:
  category_id = 9, 吃饭

用户C:
  category_id = 15, 外食
```

家庭按分类统计时，如果直接按 `category_id` 聚合，会变成三个分类。

所以分类也需要类似设计：

```text

  id
  user_id
  name
  type
  standard_category_code
  category_group_key
```

标准分类示例：

```text

TRANSPORT
SHOPPING
HOUSING
MEDICAL
EDUCATION
SALARY
INVESTMENT
TRANSFER
EXCHANGE
OTHER
```

家庭分类统计使用：

```text
standard_category_code 或 category_group_key
```

个人明细仍然使用：

```text
category_id
```

### 8.9 家庭统计视角

### 8.11 Category 与 CategoryGroup 聚合设计

决策：不再单独增加 `OperationType`。收入 / 支出下的具体统计维度统一由 `Category` 承担。

但家庭是多用户场景，不同用户的分类 ID 和分类名称可能不同，因此仍然需要增加一个标准化聚合维度：

```text
Category = 用户自己的分类
CategoryGroup = 家庭 / 系统统计用的聚合分类
```

例如：

```text
用户A:
  category_id = 1, 餐饮 -> category_group = FOOD
  category_id = 2, 外卖 -> category_group = FOOD
  category_id = 3, 买菜 -> category_group = GROCERY

用户B:
  category_id = 8, 吃饭 -> category_group = FOOD
  category_id = 9, 下馆子 -> category_group = FOOD
  category_id = 10, 超市 -> category_group = GROCERY
```

家庭统计时：

```text
GROUP BY category_group_id
```

结果：

```text
FOOD 餐饮：用户A 餐饮 + 外卖，用户B 吃饭 + 下馆子
GROCERY 买菜日用：用户A 买菜，用户B 超市
```

### 8.12 为什么不直接叫 family_category

第一阶段不建议直接叫 `family_category`，而建议叫：

```text
category_group
```

原因：

```text
它不只服务家庭统计，也可以服务个人统计、系统默认统计、账单导入推荐和图表展示。
```

如果未来确实需要每个家庭自定义自己的统计口径，再扩展：

```text
family_category_group
family_category_mapping
```

例如：

```text
家庭A：奶粉 / 玩具 / 早教 -> 育儿
家庭B：玩具 -> 娱乐
```

第一阶段先采用全局标准 `category_group`，复杂度更低。

### 8.18 家庭聚合设计结论

家庭统计不要用每个用户自己的 `account_id` 聚合，而要给账户增加标准化的聚合键，例如：

```text
provider_code
account_group_key
account_type
```

个人明细用：

```text
account_id
```

家庭汇总用：

```text
account_group_key
```

如果后面引入 Bucket，也遵循同样原则：

```text
个人余额看 bucket_id
家庭余额聚合看 account_group_key + bucket_type + currency
```

分类也同理：

```text
个人明细看 category_id
家庭统计看 category_group_key / standard_category_code
```

收入 / 支出下的具体操作类型也同理：

```text
个人明细看 operation_type_id
家庭统计看 operation_type_group_key / standard_operation_type_code
```

### 9.6 channel_template 表

### 9.8 financial_event 表

交易主表，记录业务事件。

```text
financial_event
  id
  user_id
  event_group_id 可选
  event_type
  description
  category_id
  channel_type
  channel_account_id 可选
  event_time
  currency
  amount
  base_currency
  base_amount
  exchange_rate
  include_in_statistics
  source
  remark
  created_at
  updated_at
```

字段说明：

```text
event_group_id：可选，用于关联同一业务动作产生的多笔交易，例如换汇 + 手续费、转账 + 手续费、投资买入 + 手续费。
channel_type：交易渠道编码，对应 channel_template.channel_code。收入时 UI 展示为收款渠道，支出时 UI 展示为支付渠道。
channel_account_id：可选，关联用户自己的渠道账户，例如微信账户、支付宝账户。
```

决策：`financial_event` 不保存 `primary_bucket_id`。Bucket 相关信息统一从 `ledger_entry` 获取。

原因：

```text
LedgerEntry 是余额变化的唯一来源
避免 financial_event.primary_bucket_id 和 ledger_entry.bucket_id 不一致
转账、换汇、多流水场景更统一
```

收入 / 支出列表需要展示入账桶或出账桶时，由后端根据该交易下的 `ledger_entry.amount` 正负号组装返回。

```text

expense

exchange_out
exchange_in
fee
receivable_create
receivable_collect
investment_buy
```

### 10.2.1 入账桶 / 出账桶 与 渠道的区别

核心规则：

```text
入账桶 / 出账桶：决定 Bucket 余额怎么变化
收款渠道 / 支付渠道：只记录交易通过什么方式发生
```

因此：

```text
余额统计看 bucket / ledger_entry
渠道统计看 financial_event.channel_type
```

例如同样是微信相关交易：

```text
微信零钱支付：出账桶 = 微信零钱，支付渠道 = 微信
微信绑定银行卡支付：出账桶 = 银行卡，支付渠道 = 微信
微信收款进入零钱：入账桶 = 微信零钱，收款渠道 = 微信
微信收款结算到银行卡：入账桶 = 银行卡，收款渠道 = 微信
```

### 10.2.2 数据字段映射

UI 字段可以映射到 FinancialEvent + LedgerEntry：

```text
收入：
  入账桶 -> ledger_entry.bucket_id，direction = in
  收款渠道 -> financial_event.channel_type

支出：
  出账桶 -> ledger_entry.bucket_id，direction = out
  支付渠道 -> financial_event.channel_type
```

LedgerEntry 生成规则：

```text
收入：根据入账桶生成 direction = in 的 ledger_entry
支出：根据出账桶生成 direction = out 的 ledger_entry
```

`channel_type` 不直接影响余额。Bucket 余额只由 `ledger_entry` 影响。

```text

```

推荐当前系统优先使用：


## 11. 推荐分阶段实现路线

不要一次性全做，建议分 4 个阶段。

### 阶段 1：多币种基础能力

目标：让现有收入 / 支出支持币种。

工作内容：

1. 增加币种字段：
   - `currency`
   - `base_currency`
   - `base_amount`
   - `exchange_rate`
2. 用户增加默认基准币种，默认 CNY。
3. 新增汇率表或汇率配置。
4. 所有统计统一使用 `base_amount`。
5. 前端新增币种选择。

这一阶段可以暂时不做 Bucket。

### 阶段 2：账户余额桶 Bucket

目标：能记录“微信里有多少钱”。

工作内容：

1. 新增 Bucket。
2. Account 和 Bucket 建立一对多关系。
3. 交易记录绑定 Bucket。
4. 收入增加 Bucket 余额。
5. 支出减少 Bucket 余额。
6. 支持余额调整。
7. 建议引入或预留 `ledger_entry`。

### 阶段 3：换汇和转账

目标：处理非收入 / 支出的资金流动。

工作内容：

1. 增加交易类型：
   - transfer
   - exchange
2. 换汇记录 from_bucket 和 to_bucket。
3. 换汇本金不进入收入支出统计。
4. 手续费可以计入支出。
5. 更新两个 Bucket 的余额。

### 阶段 4：投资模块

目标：投资收益 / 亏损和普通账单分离。

工作内容：

1. 新增投资资产表。
2. 新增投资快照表。
3. 投资收益 / 亏损曲线从投资快照读取。
4. 历史投资账单迁移到投资快照。
5. 普通收入支出统计排除投资损益。

## 12. 关键设计决策

### 决策 1：Bucket 是否和 Account 绑定？

建议：

```text
绑定，但不要等同。
```

关系：

```text
Account 1 : N Bucket
```

理由：

- 支持一个账户多个币种。
- 支持一个账户多个资金池。
- 统计时可以按账户汇总。
- 余额管理更清晰。

### 决策 2：换汇算不算收入支出？

建议：

```text
不算。
```

理由：

- 换汇只是资产形态变化。
- 不应该污染收入 / 支出曲线。
- 手续费可以算支出。

### 决策 3：投资收益是否继续放账单表？

建议：

```text
长期不要。
```

理由：

- 投资收益不是普通收入。
- 浮盈浮亏不是现金流。
- 投资需要本金、市值、收益率等独立维度。
- 投资曲线图应该独立统计。

短期可以兼容现有数据。

### 决策 4：是否需要 Ledger？

建议：

```text
如果要做余额，最好需要。
```

理由：

- 余额需要可追溯。
- 删除 / 修改交易需要重算。
- 换汇 / 转账天然是多条余额变动。
- 投资买入卖出也会影响现金桶。

### 决策 5：多币种统计用哪个金额？

建议：

```text
统计默认用 base_amount。
展示明细用 amount + currency。
```

例如：

```text
明细：100 USD
统计：720 CNY
```

## 13. 最终推荐方案

如果要确定一个长期稳定的设计方向，建议采用：

1. 保留现有 `financial_event` 表，但扩展为业务事件表。
2. 新增 `bucket` 表，余额从 Bucket 管理。
3. 新增 `ledger_entry` 表，所有影响余额的交易都生成流水。
4. 新增 `exchange_detail` 表，保存换汇详情。
5. 新增 `investment_asset` 和 `investment_snapshot` 表，投资单独建模。
6. 普通收入 / 支出统计默认只统计：

```text
收入
支出
```

排除：

```text
换汇
转账
投资买入
投资卖出
投资浮盈
投资浮亏
余额调整
```

## 14. 最推荐的第一步

第一步不建议直接做完整投资模块，也不建议直接做复杂换汇。

建议优先做：

```text
多币种 + Bucket 基础模型
```

第一阶段目标：

> 每条收入 / 支出账单都支持币种，并且可以选择进入哪个余额桶；系统能够展示每个桶当前余额，以及按基准币种统计收入支出曲线。

完成后再依次做：

```text
换汇
  ↓
投资模块
```

## 15. 一句话总结

这次升级的核心不是“给记账系统加几个字段”，而是把系统升级为：

> 一个支持多币种、多账户余额、换汇、投资资产表现的个人 / 家庭财务管理系统。

设计上应该从「收入 / 支出账单」扩展为：

```text
业务交易 FinancialEvent
  +
余额流水 LedgerEntry
  +
资产快照 InvestmentSnapshot
```
