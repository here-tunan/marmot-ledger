# 导入识别与规则学习设计方案

## 1. 目标

当前系统大量依赖微信、支付宝等账单文件导入。后续系统会升级为更完整的 Financial App，引入：

```text
FinancialEvent
LedgerEntry
Bucket
CategoryGroup
ChannelTemplate
Receivable
Investment
Deposit
```

如果每一笔导入记录都需要手动选择分类、桶、交易类型，会非常麻烦。

因此需要一个导入识别系统：

```text
普通交易自动处理
不确定交易进入待确认
复杂交易给出候选建议
用户确认后沉淀为规则
系统越用越准
```

---

## 2. 总体思路

导入识别不应该只靠精确匹配，也不应该一开始完全依赖 AI。

推荐采用分层识别：

```text
第 1 层：确定性规则
第 2 层：关键词 / 模糊规则
第 3 层：商户名称归一
第 4 层：历史统计推荐
第 5 层：AI 辅助建议
```

第一阶段先实现：

```text
规则层：equals / contains
归一层：merchant_alias
统计层：merchant_category_stats
```

第二阶段再考虑：

```text
AI 建议层
AA / 押金 / 投资复杂交易识别
```

---

## 3. 导入处理流程

完整流程：

```text
1. 用户导入微信 / 支付宝 / 银行账单文件
2. 系统保存原始导入记录 import_record
3. 系统运行识别规则，生成 financial_event_draft
4. 根据置信度决定自动确认还是待确认
5. 用户确认或修改草稿
6. 系统生成正式 financial_event + ledger_entry
7. 用户确认结果沉淀成规则
8. 下次导入时复用规则
```

---

## 4. 识别状态

每条导入记录可以有以下状态：

```text
auto_confirmed      自动确认
pending_confirm     待用户确认
needs_split          需要拆分 / 复杂处理
possible_duplicate   疑似重复
ignored              已忽略
```

### 4.1 自动确认

当系统非常确定时自动入账。

条件示例：

```text
交易类型明确
分类明确
Bucket 明确
渠道明确
没有复杂交易标记
没有疑似重复
置信度 >= 80
```

### 4.2 待确认

当系统能给出建议，但缺少关键字段或置信度不够。

例如：

```text
分类不确定
Bucket 不确定
商户首次出现
关键词容易歧义
```

### 4.3 需要拆分

当一条或多条记录可能属于复杂业务。

例如：

```text
AA / 群账单
押金
报销
基金定投
投资买入
退款
大额转账
```

## 5. 核心数据表

## 5.1 import_record 原始导入记录

保存微信 / 支付宝 / 银行 CSV 的原始数据。

```text
import_record
  id
  user_id
  source
  trade_time
  trade_type
  merchant
  counterparty
  description
  amount
  direction
  payment_method
  status
  raw_data
  created_at
```

字段说明：

```text
source：WECHAT / ALIPAY / BANK
merchant：商户名称
counterparty：交易对方
payment_method：支付方式，例如 工商银行储蓄卡(2781)、零钱
raw_data：原始行数据，方便回溯和重新识别
```

---

## 5.2 financial_event_draft 交易草稿

系统识别后先生成草稿，不一定立刻入正式账。

```text
financial_event_draft
  id
  import_record_id
  user_id
  suggested_event_type
  suggested_category_id
  suggested_category_group_id
  suggested_bucket_entries
  suggested_channel_code
  confidence_score
  status
  risk_flags
  created_at
  updated_at
```

`suggested_bucket_entries` 可以用 JSON 保存：

```json
[
  {
    "bucket_id": 1,
    "amount": -11.40
  }
]
```

`risk_flags` 示例：

```text
complex_candidate
possible_duplicate
missing_bucket
missing_category
large_amount
```

---

## 5.3 import_mapping_rule 导入映射规则

这是规则学习的核心表。

```text
import_mapping_rule
  id
  user_id
  source
  rule_type
  rule_name
  match_field
  match_type
  match_value
  target_event_type
  target_category_id
  target_category_group_id
  target_bucket_id
  target_channel_code
  priority
  confidence
  auto_confirm
  enabled
  created_from_import_record_id
  created_at
  updated_at
```

### rule_type

```text
category           分类规则
bucket             Bucket 匹配规则
event_type   交易类型规则
channel            渠道规则
complex_candidate 复杂交易候选规则
ignore             忽略规则
```

### match_field

```text
merchant
counterparty
description
payment_method
trade_type
amount
```

### match_type

第一阶段支持：

```text
equals      完全相等
contains    包含
```

后续可扩展：

```text
starts_with
regex
amount_range
```

### priority

建议优先级：

```text
用户手动规则：100
用户历史学习规则：90
系统内置商户规则：60
系统关键词规则：40
默认规则：10
```

---

## 5.4 merchant_alias 商户归一表

用于处理同一商户的不同名称。

```text
merchant_alias
  id
  user_id
  source
  raw_merchant_name
  normalized_merchant_name
  created_at
  updated_at
```

示例：

```text
luckin coffee -> 瑞幸咖啡
LUCKIN COFFEE -> 瑞幸咖啡
瑞幸咖啡杭州印象城店 -> 瑞幸咖啡
```

归一后，分类规则可以绑定：

```text
normalized_merchant_name = 瑞幸咖啡
```

而不是绑定每一个原始商户名。

---

## 5.5 merchant_category_stats 商户分类统计表

用于根据历史确认记录做概率推荐。

```text
merchant_category_stats
  id
  user_id
  normalized_merchant_name
  category_id
  category_group_id
  count
  last_used_at
  created_at
  updated_at
```

示例：

```text
瑞幸咖啡 -> 餐饮 20 次
木木水也 -> 餐饮 8 次
陈伯华 -> 买菜 5 次
```

如果某个商户历史上 90% 以上都被归为同一分类，可以作为高置信度推荐。

## 6. 规则示例

### 6.1 商户分类规则

```text
source = WECHAT
rule_type = category
match_field = merchant
match_type = equals
match_value = luckin coffee
target_category_id = 餐饮
priority = 100
auto_confirm = true
```

含义：

```text
微信账单中，商户等于 luckin coffee，则分类为餐饮。
```

---

### 6.2 支付方式到 Bucket 的规则

```text
source = WECHAT
rule_type = bucket
match_field = payment_method
match_type = equals
match_value = 工商银行储蓄卡(2781)
target_bucket_id = 工商银行-CNY
priority = 100
auto_confirm = true
```

含义：

```text
微信账单中支付方式为工商银行储蓄卡(2781)，则出账 Bucket 为工商银行-CNY。
```

---

### 6.3 关键词分类规则

```text
source = WECHAT
rule_type = category
match_field = description
match_type = contains
match_value = 羽毛球
target_category_id = 运动
priority = 80
auto_confirm = false
```

含义：

```text
说明包含“羽毛球”，建议分类为运动。
```

但不自动确认，因为羽毛球场地可能涉及 AA。

---

### 6.4 投资候选规则

```text
source = ALIPAY
rule_type = event_type
match_field = description
match_type = contains
match_value = 基金定投
target_event_type = investment_buy
priority = 90
auto_confirm = false
```

含义：

```text
包含“基金定投”的记录，建议为投资买入，需要用户确认投资 Bucket。
```

## 7. 识别算法

### 7.1 处理步骤

```text
1. 根据 source 推断 channel_code
2. 根据金额方向初步判断收入 / 支出
3. 加载用户规则和系统规则
4. 匹配规则
5. 按 priority 从高到低应用规则
6. 尝试 merchant_alias 归一
7. 查询 merchant_category_stats 历史概率
8. 计算 confidence_score
9. 判断是否自动确认
```

---

### 7.2 伪代码

```go
func BuildDraft(record ImportRecord, userID int64) FinancialEventDraft {
    draft := FinancialEventDraft{
        ImportRecordID: record.ID,
        UserID:         userID,
        Status:         "pending_confirm",
    }

    draft.SuggestedChannelCode = inferChannel(record.Source)
    draft.SuggestedFinancialEventType = inferFinancialEventType(record)

    normalizedMerchant := NormalizeMerchant(record.Merchant, userID, record.Source)

    rules := LoadEnabledRules(userID, record.Source)
    matchedRules := MatchRules(record, normalizedMerchant, rules)
    SortByPriority(matchedRules)

    for _, rule := range matchedRules {
        ApplyRule(&draft, rule)
    }

    ApplyMerchantStatsSuggestion(&draft, userID, normalizedMerchant)

    draft.ConfidenceScore = CalculateConfidence(draft, matchedRules)

    if CanAutoConfirm(draft, matchedRules) {
        draft.Status = "auto_confirmed"
    }

    return draft
}
```

## 8. 匹配实现

### 8.1 字段归一化

匹配前先做 normalize：

```text
去空格
转小写
中英文括号统一
去掉金额符号
去掉多余标点
```

示例：

```text
工商银行储蓄卡（2781）
工商银行储蓄卡(2781)
```

归一后应视为相同。

---

### 8.2 匹配函数伪代码

```go
func MatchRule(record ImportRecord, rule ImportMappingRule) bool {
    value := GetFieldValue(record, rule.MatchField)

    switch rule.MatchType {
    case "equals":
        return normalize(value) == normalize(rule.MatchValue)
    case "contains":
        return strings.Contains(normalize(value), normalize(rule.MatchValue))
    default:
        return false
    }
}
```

## 9. 置信度设计

第一阶段可以简单打分。

加分项：

```text
分类命中用户规则 +40
Bucket 命中用户规则 +30
交易类型明确 +20
渠道明确 +10
历史分类概率 >= 90% 且次数 >= 3 +30
商户归一成功 +10
```

扣分项：

```text
复杂关键词 -30
疑似重复 -50
缺少 Bucket -30
缺少分类 -20
大额交易 -10
```

自动确认建议阈值：

```text
confidence_score >= 80
```

---

## 10. 自动确认规则

自动确认需要同时满足：

```text
交易类型明确
分类明确
Bucket 明确
渠道明确
没有复杂交易标记
没有疑似重复
所有关键规则允许 auto_confirm
confidence_score >= 80
```

复杂交易不建议自动确认：

```text
AA / 群收款
押金 / 保证金
基金 / 理财 / 证券
退款
报销
大额转账
跨币种交易
```

这些只做候选建议。

如果用户勾选，系统写入 `import_mapping_rule`。

## 16. 总结

导入识别系统不是单纯精确匹配，而是：

```text
规则层 + 商户归一层 + 历史统计层 + AI 建议层
```

第一阶段先实现：

```text
用户规则
关键词规则
商户归一
历史统计推荐
待确认队列
规则管理
```

这样可以做到：

```text
普通交易自动处理
复杂交易给出候选建议
用户确认后系统持续学习
导入越多，识别越准确
```
